## **프로젝트 방향성**

프로젝트를 시작하기 전에 개발업계에 종사하는 친구, 학원 강사님, 인턴 시절 사수분들께 실무에 들어가기 전 어떤 준비를 해야할지, 포트폴리오에 중점적으로 담아야할 부분은 어떤 것인지 자문을 구했고 본 프로젝트의 컨셉을 정했습니다.

### 1. 사용한 코드는 완벽하게 이해

구글링, chat GPT와 같은 인공지능 기술들이 빠른 속도로 발전하고 있습니다.

즉 모든 정답은 이미 인터넷에 나와있습니다. 

하지만 실무에서 솔루션코드를 단순히 복사,붙여넣기하여 코드를 진행하면 문제가 발생했을 때, 대처능력이 현저히 떨어지게 될 것입니다. 

이는 실무에서 치명적인 결과로 이어질 것이고 개인 성장 측면에서도 문제 해결 기술을 배우고 성장할 수 있는 기회를 놓치는 것이라 생각합니다.  

본 프로젝트를 실제 프로젝트라고 생각하고 구글링을 하되 외부 정보는 참고로만 사용하고 내가 사용할 코드는 완벽하게 이해하고 직접 하나하나 타이핑할 것입니다. 

제가 이해하지 못하는 코드를 마주치면 우회해야할 장벽이 아니라 성장 할 수 있는 기회라 생각하고 프로젝트에 전념하겠습니다.

### 2. 다양성보다는 깊이 & 필수적이고 기본적인 기술(CRUD)

이 프로젝트에서 사용하는 기술들은 실무에서 사용하지 않을 기술들도 있을 것이고

무엇보다 실무에서 사용하는 기술을 저 혼자 공부해서 적용하는 것은 힘들 것입니다.

본 프로젝트에서는 백엔드 개발자로서 필수적이고 기본이라 여겨지는 CRUD 작업을 진행함과 동시에 프로젝트의 진행과정을 문서화하여 새로운 지식을 쌓는 방식과 태도를 보여드리고자 합니다.

~~‘이렇게 많은 기술, 기능을 써봤습니다’~~ 가 아닌

**‘한가지 문제를 해결하기 위해 깊게 들어가봤습니다’** 를전달하고자 합니다.

**‘즉 하나를 하더라도 완벽하게 원리를 이해하겠습니다’**. 

회사에 입사하게 되었을 때 모든 작업과 학습기회에 같은 마음가짐과 태도를 유지하겠다는 약속을 드리겠습니다. 

### 3. 타인이 보기 편한 구성 - 클린 코드

코드는 저만 이해하면 되는 것이 아닌 타인이 이해할 수 있어야 합니다.

회사에 들어가더라도 신입으로서 전 제 코드를 상사분들께 평가받게 될 것입니다.

타인이 제가 쓴 코드를 봤을 때 비일관성, 비효율성, 목적을 명확하게 나타내지 않는 변수명을 보면 

불편함을 느끼실 것이라 생각합니다.

 

실무에서도 마찬가지로 저 혼자 하는 것이 아닌 타인과 공유를 하면서 진행하는 것이기에 

본 프로젝트를 실제 프로젝트라 생각하고 추가 설명없이 코드만 봐도 이해할 수 있게 하겠습니다.

실무라고 가정하면 이는 팀원분들의 시간과 노력을 절약시키고, 업무 능률 상승으로 이어질 것이라 믿습니다.

## **프로젝트 원칙**

- 변수 이름은 명확성을 띌 것

- GIT Commit 규칙을 지킬 것
    
    feat : 새로운 기능 추가, 기존의 기능을 요구 사항에 맞추어 수정
    fix : 기능에 대한 버그 수정
    build : 빌드 관련 수정
    chore : 패키지 매니저 수정, 그 외 기타 수정 ex) .gitignore
    ci : CI 관련 설정 수정
    docs : 문서(주석) 수정
    style : 코드 스타일, 포맷팅에 대한 수정
    refactor : 기능의 변화가 아닌 코드 리팩터링 ex) 변수 이름 변경
    test : 테스트 코드 추가/수정
    release : 버전 릴리즈
    

- 내가 쓴 코드들은 내가 완벽하게 이해하고 직접 타이핑 할 것(복사&붙여넣기 금지)
